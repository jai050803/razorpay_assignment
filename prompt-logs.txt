PROMPT 2 — Database Config & Migration Files

Create the PostgreSQL database config and all migration files for this project.
 
In src/config/db.js:
- Export a pg Pool using environment variables from .env
- The pool should be used across the whole app (singleton pattern)
 
Create four migration SQL files in migrations/ folder (run in order):
 
migrations/001_create_users.sql:
- Table: users
- Columns: id (SERIAL PK), name (VARCHAR 255 NOT NULL), email (VARCHAR 255 UNIQUE NOT NULL),
  password (VARCHAR 255 NOT NULL), role (VARCHAR 10, DEFAULT 'EMP'), created_at (TIMESTAMP DEFAULT NOW())
- Valid roles will be: EMP, RM, APE, CFO
 
migrations/002_create_employee_rm_assignments.sql:
- Table: employee_rm_assignments
- Columns: id (SERIAL PK), emp_id (INT FK → users.id), rm_id (INT FK → users.id),
  created_at (TIMESTAMP DEFAULT NOW())
- Constraint: UNIQUE(emp_id) — each EMP reports to exactly one RM
 
migrations/003_create_reimbursements.sql:
- Table: reimbursements
- Columns: id (SERIAL PK), user_id (INT FK → users.id), title (VARCHAR 255 NOT NULL),
  description (TEXT NOT NULL), amount (NUMERIC 12,2 NOT NULL),
  status (VARCHAR 20 DEFAULT 'PENDING'), rm_approved (BOOLEAN DEFAULT FALSE),
  ape_approved (BOOLEAN DEFAULT FALSE), created_at (TIMESTAMP DEFAULT NOW()),
  updated_at (TIMESTAMP DEFAULT NOW())
 
migrations/004_create_reimbursement_approvals.sql:
- Table: reimbursement_approvals (audit log)
- Columns: id (SERIAL PK), reimbursement_id (INT FK → reimbursements.id),
  approver_id (INT FK → users.id), approver_role (VARCHAR 10 NOT NULL),
  action (VARCHAR 20 NOT NULL), created_at (TIMESTAMP DEFAULT NOW())
 
In scripts/migrate.js:
- Read all .sql files from migrations/ folder, sorted alphabetically
- Execute them in order using the pg pool
- Log success or failure for each file
- This is what npm run db:migrate calls

prompt 4 : Auth Middleware (Cookie Session)

Set up stateless cookie-based JWT authentication for the application.

In src/app.js:
- Import and use 'cookie-parser' middleware globally before any routes.

In src/utils/constants.js:

- Export STATUSES = { PENDING: 'PENDING', APPROVED: 'APPROVED', REJECTED: 'REJECTED' }

In src/utils/response.js:
- Export successResponse(res, data, statusCode=200) — sends JSON { status: 'success', data }
- Export errorResponse(res, message, statusCode=400) — sends JSON { status: 'error', message }

In src/utils/validators.js:
- Export isOrgEmail(email) — returns true if email explicitly ends with '@org.com'

In src/middleware/auth.middleware.js:
- Export a function 'requireAuth' that:
  1. Reads the authentication cookie (e.g., named 'token') from req.cookies using cookie-parser.
  2. If the cookie does not exist, immediately returns a 401 status using errorResponse(res, 'Unauthorized').
  3. If it exists, verifies the JWT token using a JWT_SECRET environment variable (fallback to a secret string if not in .env).
  4. Decodes the user ID from the token payload, fetches the full user profile from the database, and attaches it directly to req.user.
  5. Calls next() if the user is verified and exists. If token verification fails or the user no longer exists in the DB, clear the cookie and return a 401 error.

In src/middleware/role.middleware.js:
- Export a factory function 'requireRole(...roles)' that:
  1. Returns an Express middleware function.
  2. Checks if the current authenticated user's role (req.user.role) is included in the allowed roles array.
  3. If not included, returns a 403 status using errorResponse(res, 'Forbidden').
  4. If included, calls next().

Keep the code clean, modular, properly structured according to SOLID principles, and free of any TypeScript code.

prompt 5: Onboarding Module (Register, Login, Logout)

 
Build the onboarding module. These are PUBLIC endpoints (no auth required).
 
In src/modules/onboarding/onboarding.service.js:
- registerUser({ name, email, password }):
  - Validate all fields are present
  - Validate email is @org.com domain
  - Check email is not already taken → 409 if taken
  - Hash password with bcrypt (10 rounds)
  - Insert user into DB with role = 'EMP'
  - Return the created user (without password)
 
- loginUser({ email, password }):
  - Validate fields are present
  - Validate email is @org.com domain
  - Find user by email → 401 if not found
  - Compare password with bcrypt → 401 if wrong
  - Return the user (without password)
 
In src/modules/onboarding/onboarding.controller.js:
- register(req, res): calls service, sets req.session.userId on success
- login(req, res): calls service, sets req.session.userId on success
- logout(req, res): destroys req.session, clears cookie
 
In src/modules/onboarding/onboarding.routes.js:
- POST /register → register controller
- POST /login → login controller
- POST /logout → logout controller
- These will be mounted under /rest/onboardings in app.js
 
Error format for all failures:
{ "status": "error", "message": "descriptive error message" }
 
Success format for register/login:
{ "status": "success", "data": { "userId": ..., "name": ..., "email": ..., "role": ... } }
